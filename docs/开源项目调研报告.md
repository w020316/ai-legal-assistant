## 三、各项目优势与不足深度分析

### 3.1 LaWGPT（pengxiao-song/LaWGPT）

**优势**：
- 中文法律大模型先驱，基于 Chinese-Llama + 35 万法律问答数据集微调
- 支持法律问答、案例分析、判决意见生成、法律概念解释四大场景
- PEFT 轻量级优化技术，单 GPU 可运行，本地部署门槛低
- Apache-2.0 协议商用友好

**不足**：
- 2023-05 后停更，未跟进最新模型架构
- 仅 Gradio 界面，无 Web 应用工程化设计
- 无 RAG 检索增强、无合同审查、无文书模板生成
- 案例分析仅生成案情描述，不引用真实法条

**与 linzAI 契合度**：⭐⭐ 低。LaWGPT 是模型层项目，linzAI 已用 API 路由方案，无需自训模型。其法律问答场景 linzAI 已覆盖且更完善（RAG + SSE + 引用溯源）。

### 3.2 ChatLaw（PKU-YuanGroup/ChatLaw）

**优势**：
- 北大团队主导，MoE 混合专家架构（4×7B），法律任务动态路由到专业子模型
- 多智能体协作系统：Legal Assistant → Legal Researcher → Senior Lawyer，模拟律所工作流
- 自研 Text2Vec 文本相似度模型，93000 案件训练，案例检索能力强
- 法律知识图谱增强，提升推理逻辑性
- LawBench 评测准确率超 GPT-4 7.73%
- Self-Suggestion 自我监督机制降低幻觉
- MIT 协议商用友好

**不足**：
- 2023-07 后低活跃，社区维护力度下降
- 部署需 GPU（推荐 12GB+ 显存），硬件成本高
- 录音多模态偏向语音输入，无图片识别
- 无合同审查、无安全评分、无条款通俗化
- Python 技术栈与 linzAI（Java）差异大

**与 linzAI 契合度**：⭐⭐⭐ 中。多智能体协作架构和 Text2Vec 案例检索思路可借鉴，但模型层方案 linzAI 无需复刻。

### 3.3 DISC-LawLLM（FudanDISC/DISC-LawLLM）

**优势**：
- 复旦 DISC 实验室学术背书，Baichuan-13B 全参指令微调
- **法律三段论 prompting**：以"法官基本法律推理过程"重构指令数据，显著提升推理能力
- 30 万高质量 SFT 数据集（DISC-Law-SFT）开源
- 配套评测基准 DISC-Law-Eval，客观+主观双维度评测
- 检索增强模块，支持外部法条/案例调用
- 服务法律专业者（文书摘要/事件检测/实体关系抽取）+ 普通民众（法律咨询）+ 法学生（考试辅导）三类用户

**不足**：
- 学术项目，2023-09 后低活跃
- CC BY-SA 4.0 协议，商用需同等开源
- 无 Web 应用，仅 Gradio Demo
- 无合同审查、无安全评分等业务功能
- 部署需 13B 模型显存

**与 linzAI 契合度**：⭐⭐⭐ 中。法律三段论 prompting 策略可借鉴用于优化 linzAI 的 LEGAL_QA_SYSTEM prompt，提升推理严谨性。DISC-Law-Eval 评测思路可参考建立 linzAI 质量基线。

### 3.4 Mike（willchen96/mike）

**优势**：
- **中国开发者主导，技术栈与 linzAI 最接近**（TypeScript + Next.js + Supabase + PostgreSQL + pgvector）
- 双模式合同智能审查：单份风险标注 + 双合同差异比对
- 本地化文档理解引擎：OCR + PDF 解析 + LibreOffice 转换，处理扫描件/加密 PDF/老旧 DOC
- 私有化大模型接入框架：OpenAI/Ollama/DeepSeek/Qwen 多模型适配
- Supabase 驱动的法律知识库：条款库/案例索引/法规标签，支持团队协作标注与版本回溯
- 自动化文档工作流：邮件收合同 → 存 R2 → AI 初审 → 推送飞书/企微 → 生成修订建议 → 归档 PostgreSQL
- AGPL-3.0 但支持私有部署，等保 2.0 与《个人信息保护法》合规
- 2026-05 仍活跃维护

**不足**：
- AGPL-3.0 协议要求衍生作品开源，商用受限
- 无安全评分、无条款通俗化、无缺失条款检测（这些 linzAI v1.4.3 已实现）
- 无法律问答、无案例检索、无文书模板生成
- 依赖 Supabase + Cloudflare R2，迁移成本高

**与 linzAI 契合度**：⭐⭐⭐⭐ 高。技术栈高度相似，**双合同比对**和**自动化文档工作流**是最值得借鉴的功能。OCR + LibreOffice 文档解析能力可补强 linzAI 当前仅支持 PDF/DOCX 的局限。

### 3.5 AI Legal Assistant（1.1k star 多 Agent 项目）

**优势**：
- **5 Agent 并行协同**：条款分析师、风险评估师、合规检查师、条款映射师、建议引擎，60 秒生成综合报告
- **安全评分 + A-F 等级**：0-100 分量化合同风险
- **风险财务估算**：量化高风险条款的潜在经济损失
- **条款通俗化翻译**：法律术语→大白话
- **缺失保护条款检测**：扫描合同漏洞（终止费/不可抗力等）
- **义务时间线**：梳理截止日期与违约后果
- **谈判建议优先级**：TIER 1-WALK AWAY / TIER 2-PUSH HARD
- 14 种指令覆盖合同审核/文档生成/合规审计三大板块
- 支持 GDPR/CCPA/PCI-DSS 合规框架

**不足**：
- 无公开 GitHub 仓库地址，复刻难度高
- Python 技术栈，与 linzAI（Java）差异大
- 无 Web 界面，依赖指令式交互
- 无案例检索、无知识库管理
- 无中文法律适配（偏英美法系）

**与 linzAI 契合度**：⭐⭐⭐⭐⭐ 最高。linzAI v1.4.3 已借鉴其**安全评分、风险分级、条款通俗化、缺失条款检测**四大特色。剩余的**风险财务估算、义务时间线、谈判建议优先级、双合同比对、合规审计**是下一阶段最值得借鉴的方向。

## 五、linzAI 下一阶段实现方案

### 5.1 第一阶段（v1.5.0）：合同审查深化

**目标**：借鉴 AI Legal Assistant，补齐合同审查报告的专业维度。

| 功能 | 实现方案 | 涉及文件 |
|------|----------|----------|
| 双合同比对 | DocumentService 新增 compare(userId, docId1, docId2) 方法，AI prompt 输出差异条款 + 风险变化 JSON | DocumentService.java + DocumentController.java + DocumentView.vue |
| 风险财务估算 | RiskPoint 新增 financialExposure 字段，CONTRACT_REVIEW_SYSTEM prompt 要求量化经济损失 | RiskPoint.java + PromptTemplates.java + DocumentView.vue |
| 义务时间线 | DocumentAnalysisVO 新增 obligations 数组（date + obligation + consequence），AI 梳理截止日期 | DocumentAnalysisVO.java + Obligation.java + PromptTemplates.java |
| 谈判建议优先级 | RiskPoint 新增 priority 字段（TIER_1/TIER_2），前端按优先级分组展示 | RiskPoint.java + DocumentView.vue |

**技术要点**：
- 后端 DTO 扩展保持向后兼容（新字段可选）
- AI prompt 增量优化，单次调用返回完整结构化 JSON
- 前端 DocumentView 新增"合同比对"入口 tab、义务时间线时间轴组件、谈判建议优先级分组卡片

### 5.2 第二阶段（v1.6.0）：文档处理能力补强

**目标**：借鉴 Mike，补齐文档解析的鲁棒性。

| 功能 | 实现方案 |
|------|----------|

| 文档预处理 | 上传后自动统一转 PDF，便于后续分析 |

### 5.3 第三阶段（v1.7.0）：问答质量提升

**目标**：借鉴 DISC-LawLLM 与 ChatLaw，提升法律问答的推理严谨性。

| 功能 | 实现方案 |

| 评测基准 | 建立 linzAI-Law-Eval，20 道客观题 + 10 道主观题，定期回归测试 |

## 六、方案确认

本调研报告已完成，结论如下：

1. **5 个候选项目均为真实可考证的高质量法律 AI 开源项目**，覆盖大模型路线（LaWGPT/ChatLaw/DISC-LawLLM）与应用层路线（Mike/AI Legal Assistant）
